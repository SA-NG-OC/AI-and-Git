Branch

Merge
Kết hợp sự thay đổi từ các branch khác nhau lại với nhau.
Pull
Lấy các thay đổi mới nhất từ remote repository về máy.
Push
Đưa các commit từ local lên remote repository.

  • Khởi tạo Git trong một thư mục để biến nó thành Git Repository.
  • Git sẽ tạo thư mục ẩn .git để lưu trữ toàn bộ thông tin quản lý phiên bản.
  • Khi file được thêm, sửa hoặc xóa, Git đánh dấu trạng thái là Modified.
  • Chọn file muốn commit bằng lệnh git add để đưa vào Staging Area.
  • File trong Staging Area sẽ được commit, tạo snapshot lưu vĩnh viễn trong lịch sử.
  • Bạn có thể xem lại toàn bộ lịch sử commit và quay về bất kỳ commit nào trước đó.
  • Git chỉ lưu sự thay đổi (diff) giữa các phiên bản, không lưu bản sao đầy đủ — giúp tiết kiệm dung lượng.

GitHub là gì?

💡 GitHub ≠ Git. GitHub là một nền tảng web sử dụng Git làm nền tảng bên dưới.
  • GitHub là kho lưu trữ mã nguồn lớn nhất thế giới, được Microsoft mua lại năm 2018.
  • GitHub cung cấp giao diện web, quản lý issue, Pull Request, GitHub Actions, GitHub Pages và nhiều hơn nữa.
  • Các nền tảng tương tự: GitLab, Bitbucket, Gitea (self-hosted).
⚙️
Sử dụng Git

git init

Khởi tạo một Git Repository mới trong thư mục hiện tại. Git sẽ tạo thư mục ẩn .git chứa toàn bộ metadata theo dõi phiên bản.

# Khởi tạo repo mới
git init

# Khởi tạo với tên branch mặc định (khuyến nghị)
git init -b main

# Clone repo từ remote về máy
git clone https://github.com/user/repo.git

Staging Environment

Staging Area giống như phòng chờ cho các thay đổi — bạn kiểm soát chính xác file nào sẽ vào commit tiếp theo.

Lệnh Công dụng
git add <file> Stage một file cụ thể
git add --all hoặc -A Stage tất cả thay đổi (bao gồm xóa file)
git add . Stage tất cả thay đổi trong thư mục hiện tại
git status Xem trạng thái các file (staged / unstaged)
git restore --staged <file> Hủy stage một file (unstage)
git diff --staged Xem chi tiết thay đổi đã được stage

Git Commit

Một commit là điểm lưu trong lịch sử project — snapshot tại một thời điểm kèm message mô tả.

Lệnh Công dụng
git commit -m "message" Commit các thay đổi đã stage kèm message
git commit -a -m "message" Commit tất cả thay đổi (bỏ qua staging; không áp dụng cho file mới)
git commit --allow-empty -m "Start" Tạo commit rỗng (hữu ích để khởi tạo project)
git commit --no-edit Commit dùng lại message của commit trước
git commit --amend --no-edit Thêm staged changes vào commit cuối, giữ nguyên message
git log Xem lịch sử commit đầy đủ
git log --oneline Xem lịch sử commit dạng rút gọn
✅ Tip: Gõ git commit (không có -m) để mở trình soạn thảo — tiện lợi khi cần viết commit message dài, nhiều dòng.

Git Branch

Branch cho phép phát triển tính năng, sửa lỗi hoặc thử nghiệm mà không ảnh hưởng đến nhánh chính.

Lệnh Công dụng
git branch Liệt kê các branch hiện có
git branch <tên> Tạo branch mới
git switch <tên> Chuyển sang branch (lệnh mới, thay thế checkout)
git switch -c <tên> Tạo và chuyển sang branch mới cùng lúc
git checkout -b <tên> Tạo và chuyển sang branch mới (lệnh cũ)
git branch -d <tên> Xóa branch (chỉ khi đã merge)
git branch -D <tên> Xóa branch bắt buộc
git branch -m <tên mới> Đổi tên branch hiện tại 
# Tạo branch feature và chuyển sang
git switch -c feature/login

# Làm việc, commit...
git add .
git commit -m "Add login feature"

# Quay về main và merge
git switch main
git merge feature/login

Git Tagging

Tag là nhãn đánh dấu một commit quan trọng, thường dùng cho các bản phát hành (release) phần mềm.

Lệnh Công dụng
git tag v1.0 Tạo lightweight tag tại commit hiện tại
git tag -a v1.0 -m "msg" Tạo annotated tag (có tên, ngày, message — khuyến nghị)
git tag v1.1 1a2b3c4d Tag một commit cụ thể theo mã SHA
git tag Liệt kê tất cả tag
git show v1.0 Xem chi tiết một tag
git push origin v1.0 Push một tag lên remote
git push --tags Push tất cả tag lên remote
git tag -d v1.0 Xóa tag ở local
git push origin --delete tag v1.0 Xóa tag trên remote
💡 Mặc định tag chỉ tồn tại trên máy local. Cần git push origin <tag> để người khác thấy được.

Git Stash

Lưu tạm thời các thay đổi chưa commit và trả về working directory sạch — hữu ích khi cần chuyển task gấp.

Lệnh Công dụng
git stash Lưu tạm thay đổi (không lưu untracked files)
git stash push -m "msg" Lưu stash kèm mô tả để dễ nhận biết
git stash -u Lưu cả untracked files
git stash -a Lưu toàn bộ kể cả ignored files
git stash list Hiện danh sách stash đã lưu
git stash show -p Xem chi tiết thay đổi của stash gần nhất
git stash apply Khôi phục stash gần nhất, giữ trong danh sách
git stash apply stash@{2} Khôi phục một stash cụ thể
git stash pop Khôi phục stash gần nhất và xóa khỏi danh sách
git stash drop stash@{1} Xóa một stash cụ thể
git stash clear Xóa toàn bộ stash
git stash branch <name> Tạo branch mới từ stash và áp dụng stash lên đó
✅ Stash lưu theo cơ chế stack (LIFO) — mỗi lần stash, thay đổi được đẩy lên đầu stack.

Git Merge

Hợp nhất lịch sử và thay đổi của một branch vào branch hiện tại.

Lệnh Công dụng
git merge <branch> Hợp nhất branch vào branch hiện tại
git merge --no-ff <branch> Luôn tạo Merge Commit (dù có thể Fast-forward)
git merge --ff-only <branch> Chỉ merge nếu có thể Fast-forward, không thì báo lỗi
git merge --squash <branch> Gộp toàn bộ commit thành một thay đổi trước khi commit
git merge --abort Hủy merge khi xảy ra conflict
git merge --continue Tiếp tục merge sau khi giải quyết conflict
⚠️ Khi xảy ra conflict, Git sẽ đánh dấu trong file bằng <<<<<<<, =======, >>>>>>>. Bạn cần sửa thủ công, rồi git add và git merge --continue.

Git Undo

git revert

git revert hoàn tác một commit cũ bằng cách tạo một commit mới — lịch sử không bị xóa, an toàn để dùng trên nhánh đã push lên remote.

Lệnh Công dụng
git revert HEAD Hoàn tác commit mới nhất bằng commit mới
git revert <commit> Hoàn tác một commit cụ thể
git revert HEAD~2 Hoàn tác commit cách hiện tại 2 bước
git revert --no-edit Hoàn tác và bỏ qua bước chỉnh sửa commit message
git log --oneline Xem lịch sử commit rút gọn để lấy mã SHA
✅ Nên dùng git revert trên nhánh đã được chia sẻ với team vì không làm thay đổi lịch sử cũ.

git reset

git reset di chuyển HEAD về một commit trước đó. Có 3 chế độ — chọn đúng chế độ để tránh mất code.

--soft

Di chuyển HEAD về commit chỉ định. Các thay đổi vẫn trong Staging Area — sẵn sàng commit lại.

--mixed (mặc định)

Di chuyển HEAD, xóa khỏi Staging Area nhưng giữ thay đổi trong Working Directory.

--hard ⚠️

Di chuyển HEAD và xóa sạch Staging Area lẫn Working Directory. Không thể hoàn tác dễ dàng!

Lệnh Công dụng
git reset --soft <commit> Giữ thay đổi trong Staging Area
git reset --mixed <commit> Giữ thay đổi trong Working Directory (mặc định)
git reset --hard <commit> Xóa toàn bộ thay đổi — không thể hoàn tác
git reset <file> Hủy stage một file cụ thể

git amend

Sửa đổi commit gần nhất: thay đổi message, thêm/bớt file, hoặc cả hai.

Lệnh Công dụng
git commit --amend Sửa commit gần nhất (mở editor để chỉnh message)
git commit --amend -m "msg mới" Thay đổi message không qua editor
git commit --amend --no-edit Thêm staged changes vào commit cuối, giữ nguyên message
⚠️ Không dùng amend trên commit đã được push lên remote — sẽ gây conflict lịch sử với team.

git reflog

Reflog là nhật ký bí mật ghi lại mọi chuyển động của HEAD — commit, checkout, reset, merge, rebase. Rất hữu ích để tìm lại commit hoặc nhánh tưởng chừng đã mất.

Lệnh Công dụng
git reflog Hiện toàn bộ lịch sử di chuyển của HEAD kèm HEAD@{n}
git reflog show <branch> Xem nhật ký của một nhánh cụ thể
git reflog --date=relative Hiện mốc thời gian tương đối (5 minutes ago...)

Git Recovery

Kết hợp git reflog với các lệnh khôi phục để cứu code trong tình huống xóa nhầm nhánh, reset sai, hoặc mất commit.

Lệnh Công dụng
git reset --hard HEAD@{n} Khôi phục về vị trí HEAD@{n} tìm được trong reflog
git reset --hard <commit_sha> Quay về commit cụ thể đã bị xóa khỏi git log
git checkout -b <tên mới> <sha> Hồi sinh nhánh đã xóa bằng cách tạo nhánh mới tại SHA cũ
git cherry-pick <sha> Lấy riêng một commit bị thất lạc và áp dụng vào nhánh hiện tại 
# Tình huống: lỡ git reset --hard, mất code
git reflog                        # tìm vị trí trước khi reset
# HEAD@{3}: commit abc1234 "Add payment feature"
git reset --hard HEAD@{3}        # khôi phục về đó

SSH là gì?

SSH (Secure Shell) là giao thức kết nối an toàn tới máy chủ từ xa. GitHub dùng SSH key thay cho mật khẩu để xác thực danh tính.

🔑
Private Key
Trên máy bạn (bí mật)
🔐
🔒
Public Key
Lưu trên GitHub
✅
☁️
Kết nối an toàn
Git push / pull
Lệnh Công dụng
ssh-keygen Tạo một cặp SSH key mới (public + private)
ssh-add ~/.ssh/id_ed25519 Thêm khóa SSH vào SSH agent
ssh-add -l Liệt kê các SSH key đã tải vào agent
ssh -T git@github.com Kiểm tra kết nối SSH tới GitHub 
# 1. Tạo SSH key (Ed25519 được khuyến nghị)
ssh-keygen -t ed25519 -C "email@example.com"

# 2. Thêm public key vào GitHub: Settings → SSH Keys → New SSH Key
cat ~/.ssh/id_ed25519.pub

# 3. Kiểm tra kết nối
ssh -T git@github.com
# Hi username! You've successfully authenticated.

Pull Request (PR)

Pull Request là cơ chế trên GitHub cho phép đề xuất thay đổi từ nhánh của bạn vào nhánh chính, kèm theo code review từ đồng nghiệp.

  • Fork repository (nếu không có quyền ghi trực tiếp) hoặc tạo branch mới trong cùng repo.
  • Thực hiện thay đổi, commit và push nhánh lên GitHub.
  • Mở Pull Request từ giao diện GitHub, mô tả rõ nội dung thay đổi.
  • Đồng nghiệp review, comment và approve hoặc yêu cầu chỉnh sửa.
  • Sau khi được approve, merge PR vào nhánh đích.
  • Xóa nhánh sau khi merge để giữ repo gọn gàng.
✅ Best practice: Viết mô tả PR rõ ràng — vấn đề gì, giải pháp ra sao, cách test — giúp review nhanh hơn.

GitHub CI/CD (GitHub Actions)

CI/CD = Continuous Integration + Continuous Deployment/Delivery. Mỗi lần push code, hệ thống tự động chạy test, build và deploy.

Lợi ích
Phát hiện bug sớm · Deploy nhanh và an toàn · Giảm thao tác thủ công · Phản hồi ngay sau mỗi push.
Workflow
Chuỗi các job chạy cùng nhau, định nghĩa bằng file .yml trong .github/workflows/.
Job
Nhóm các step chạy trên cùng một runner (máy ảo).
Step
Một tác vụ đơn lẻ: checkout code, chạy test, build Docker image...
Runner
Máy chủ thực thi job. Dùng runner của GitHub (ubuntu-latest) hoặc self-hosted.
Trigger
Điều kiện kích hoạt workflow: push, pull_request, schedule (cron), workflow_dispatch...

Ví dụ: CI Pipeline cơ bản

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:          # Chạy khi push bất kỳ branch
  pull_request:  # Chạy khi có Pull Request

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Cài Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      - name: Cài dependencies
        run: npm install

      - name: Chạy test
        run: npm test

      - name: Build project
        run: npm run build

Ví dụ: CI/CD nâng cao (nhiều job)

name: CI/CD nâng cao
on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm install
      - run: npm test

  deploy:
    runs-on: ubuntu-latest
    needs: test    # Chỉ chạy nếu job "test" thành công
    steps:
      - uses: actions/checkout@v3
      - name: Deploy lên server
        run: ./deploy.sh
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}
💡 Bỏ qua CI: Thêm [skip ci] vào commit message để không kích hoạt pipeline khi chỉ sửa tài liệu: git commit -m "Cập nhật README [skip ci]"
